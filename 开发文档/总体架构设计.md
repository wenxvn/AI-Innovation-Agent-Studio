> 项目名称：智创工坊 / AI Innovation Agent Studio  
> 英文名：Idea2MVP Agent Studio  
> 用途：作为本项目后续开发、维护、审计、重构和 AI 协作的全局参考文档。  
> 说明：本文件描述项目目标、产品范围、页面设计、技术架构、后端设计、数据模型、Agent 体系、RAG、Memory、Skill、Tool、Eval、Trace、本地启动方式、开发路线图和验收标准。

---

## 一、项目定位

本项目不是普通聊天机器人，也不是普通后台管理系统。

本项目定位为：

> **面向高校 AI 创新竞赛、创业项目孵化、科研项目设计和软件原型开发的多智能体工程平台。**

用户可以上传比赛通知、赛题 PDF、行业资料、论文、政策文件、项目想法或已有代码仓库，平台通过多智能体协作完成：

1. 赛题解析
2. 项目方向生成
3. 调研与竞品分析
4. PRD 生成
5. 系统架构设计
6. 数据库设计
7. API 设计
8. 前后端代码骨架生成
9. 测试与自动修复建议
10. 答辩稿、路演稿、README、技术文档生成
11. 上下文工程管理
12. 长期记忆管理
13. 工具调用追踪
14. Agent Trace 可视化
15. Prompt / Skill / Eval 管理

项目核心卖点：

```text
Prompt + Skill + Memory + RAG + Tool Calling + Agent Workflow + Eval + Trace
```

目标是做成一个“可观察、可调试、可评估、可沉淀”的 Agent 工程平台。

## 二、项目边界

### 2.1 第一阶段目标

第一阶段目标不是完整商业化 SaaS，而是一个可以本地运行、适合比赛答辩展示、具备核心工程结构的 MVP。

第一阶段优先级：

1. 本地一键启动。
2. 前端页面完整。
3. UI 风格统一。
4. 核心按钮有真实行为或明确 disabled。
5. Projects / Files / Memory / Skills / Outputs 至少具备基础 CRUD。
6. Agent Run 可以通过 mock 或真实 LLM 生成结构化运行记录。
7. Workflow、Context Pack、Tool Call、Eval 能可视化展示。
8. 后端 API 契约清晰。
9. 数据可持久化。
10. 可演示完整流程。

### 2.2 不强求第一阶段完成的内容

第一阶段可以暂缓：

- 真实多用户登录系统。
- 商业计费系统。
- 云端部署。
- 完整代码沙箱。
- 完整 GitHub PR 自动提交。
- 完整多模型调度。
- 大规模文档处理。
- 企业级权限系统。

### 2.3 当前部署策略

本项目当前不优先考虑 Vercel / Railway / Render / Fly.io 部署。

优先交付方式：

```text
Windows 本地一键运行
```

必须提供：

- `start.bat`
- `stop.bat`
- `docker-compose.yml`
- `.env.example`
- `README.md`

用户双击 `start.bat` 后，应能启动：

- PostgreSQL
- Redis
- MinIO 或本地文件存储
- FastAPI 后端
- Next.js 前端

目标访问地址：

```text
前端：http://localhost:3000
后端：http://localhost:8000
API Docs：http://localhost:8000/docs
```

## 五、产品核心流程

完整业务流程：

```text
创建项目
  ↓
上传赛题 / 输入想法
  ↓
文档解析与知识库构建
  ↓
Requirement Analysis Agent 解析需求
  ↓
Research Agent 调研背景与竞品
  ↓
Product Agent 生成 PRD
  ↓
Architect Agent 生成系统架构
  ↓
Coding Agent 生成代码骨架
  ↓
QA Agent 检查问题并给出修复建议
  ↓
Pitch Agent 生成答辩材料
  ↓
Outputs 保存所有产物
  ↓
Eval Dashboard 展示评分
```

核心思想：

```text
用户不是和一个聊天机器人对话，
而是在操作一个可观察的多智能体项目孵化流水线。
```

## 十一、Context Engineering 设计

每次 Agent 执行前必须构建 Context Pack，而不是把所有历史消息塞进 prompt。

Context Pack：

```json
{
  "task": "",
  "goal": "",
  "current_stage": "",
  "constraints": [],
  "project_state": {},
  "relevant_memory": [],
  "retrieved_evidence": [],
  "tool_results": [],
  "decisions_so_far": [],
  "risks": [],
  "do_not_do": [],
  "expected_output": "",
  "output_schema": {}
}
```

Context Pack Builder 流程：

1. 判断当前任务类型。
2. 选择必要项目状态。
3. 从长期记忆中选择相关偏好和历史决策。
4. 从 RAG 结果中选择高相关证据。
5. 删除重复、过期、冲突和低可信内容。
6. 汇总工具调用结果。
7. 生成结构化 Context Pack。
8. 为每条上下文标注来源。
9. 控制 token budget。
10. 输出给对应 Agent。

|---|---|
| User Memory | 用户偏好技术栈、项目风格、语言、比赛方向 |
| Project Memory | 项目目标、技术选型、架构决策、MVP 范围、里程碑 |
| Semantic Memory | 文档 chunks、论文、赛题、政策文件、代码说明 |
| Experience Memory | 有效 prompt、失败工具调用、成功修复记录、Agent 决策经验 |

Memory 必须支持：

- 创建
- 查看
- 编辑
- 删除
- 搜索
- 标记过期
- 标记可信度
- 被 Context Pack Builder 检索使用

## 十三、RAG 设计

RAG 流程：

```text
上传文件
  ↓
解析文本
  ↓
切分 chunk
  ↓
生成 embedding
  ↓
保存到 pgvector
  ↓
用户提问 / Agent 执行任务
  ↓
检索相关 chunk
  ↓
重排序
  ↓
构建 Context Pack
  ↓
生成有引用的回答
```

RAG 要求：

- 按 project_id 隔离知识库。
- 每条证据保留来源文件名、chunk_id、相似度。
- 删除文件时删除对应 chunks。
- 支持重新索引。
- 回答中可展示引用来源。

## 十五、Tool Gateway 设计

工具统一定义：

  "output_schema": {},

内置工具：

| Tool | 功能 | 是否可能需要审批 |
