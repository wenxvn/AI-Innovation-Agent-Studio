## 七、任务 1：Agent Run 完成后自动创建 Output 记录

### 7.1 背景

v1.2 报告指出：

```text
Agent Run 完成后未自动创建 Output 记录。
运行 Agent 后，生成内容只存在 agent_runs 表中，Outputs 页面看不到新产物。
```

这会导致核心业务流程断裂：

```text
输入需求 → Agent Run → 生成内容 → Outputs 保存产物
```

### 7.2 要求

修改 Agent Run 完成逻辑，使每次成功完成的 Agent Run 自动创建一条 Generated Output。

建议逻辑：

1. 在 `services/agents.py:create_agent_run()` 或等价函数中，在 Agent 成功生成结果并完成 Eval 后，调用 Outputs service 创建产物。
2. Output 类型根据 selected skill / agent intent / generated output 自动判断。
3. 至少支持以下映射：

| selected_skill / intent | output type | title 示例 |
|---|---|---|
| `prd-writer` | `prd` | `PRD - {用户输入摘要}` |
| `architecture-designer` | `architecture` | `系统架构设计 - {用户输入摘要}` |
| `idea-generator` | `idea_report` | `项目方向报告 - {用户输入摘要}` |
| `research-synthesizer` | `research_report` | `调研分析报告 - {用户输入摘要}` |
| `fastapi-generator` | `backend_code` | `FastAPI 代码骨架 - {用户输入摘要}` |
| `nextjs-generator` | `frontend_code` | `Next.js 代码骨架 - {用户输入摘要}` |
| `qa-debugger` | `test_report` | `测试与修复建议 - {用户输入摘要}` |
| `pitch-writer` | `pitch` | `答辩稿 - {用户输入摘要}` |
| 其他 | `agent_output` | `Agent 输出 - {用户输入摘要}` |

4. Output 内容应优先使用最终生成的 Markdown 文本。
5. Output 必须关联 `project_id`。
6. Output 应记录 `created_by_agent` 或等价字段。
7. 不要创建空 Output。
8. Agent Run 失败时不要创建 Output。
9. 如果创建 Output 失败，应记录错误日志和 TraceEvent，但不要吞掉异常导致前端误以为成功。

### 7.3 前端要求

1. Chat 页面运行 Agent 后，Outputs 页面刷新后能看到新产物。
2. 如果 Chat 页面当前展示产物卡片，也应保持原有展示能力。
3. 不要重复创建同一条产物。

### 7.4 验收标准

1. 新建项目。
2. 在 Chat 输入：`帮我生成这个项目的 PRD`。
3. Agent Run 成功。
4. 进入 Outputs 页面。
5. 可以看到自动生成的 PRD 产物。
6. 点击下载可以下载 Markdown。
7. 刷新页面后产物仍存在。

## 八、任务 2：Agent Run 自动创建 Tool Call 记录

### 8.1 背景

v1.2 报告指出：

```text
Tool Call 表和审批 API 已存在，但 Agent 执行过程中并未自动创建 Tool Call 记录。
Tools 页面只能展示手动/已有记录，无法反映真实 Agent 运行链路。
```

### 8.2 本轮目标

本轮不要求实现真正的 shell / GitHub / Code Sandbox 执行，但必须实现 Agent Runtime 中的 Tool Call 记录机制。

### 8.3 要求

在 Agent Run 链路中，根据实际使用的能力创建 Tool Call 记录。

至少记录以下工具调用：

| 场景 | tool_name | 是否需要审批 | 说明 |
|---|---|---:|---|
| 构建 Context Pack 时检索文档 | `rag_search` | 否 | 记录 query、命中数量、latency |
| 构建 Context Pack 时检索 Memory | `memory_search` | 否 | 记录 query、命中数量、latency |
| 创建 Output | `output_writer` | 否 | 记录 output_id、type、title |
| 未来可能写文件/删除文件 | `file_writer` / `file_delete` | 是 | 本轮可只注册，不实际执行 |
| 高风险工具占位 | `code_sandbox` / `database_write` / `github_pr` | 是 | 本轮可只在注册表中表达审批要求 |

Tool Call 字段要求：

1. `project_id` 必须正确。
2. `agent_run_id` 必须正确。
3. `tool_name` 必须明确。
4. `input` 必须记录结构化 JSON。
5. `output` 必须记录结构化 JSON。
6. `status` 至少支持 `completed` / `failed` / `pending_approval`。
7. `requires_approval` 和 `approval_status` 必须符合当前工具风险等级。
8. 失败时记录 `error`。
9. 每个 Tool Call 尽量写入 TraceEvent。

### 8.4 Tool Registry 要求

检查 `services/tools.py` 和 Tools API。

如果当前工具注册表是硬编码列表，本轮可以保留硬编码，但要做到：

1. 工具定义结构清晰。
2. 每个工具有 name、description、category、risk_level、requires_approval、input_schema。
3. 前端 Tools 页面能清楚显示哪些工具需要审批。
4. Tool Call 记录能真实出现在 Tools 页面。

### 8.5 验收标准

1. 运行一次 Agent。
2. 打开 Tools 页面。
3. 至少看到本次 Agent Run 产生的 `rag_search`、`memory_search`、`output_writer` 中的一项或多项 Tool Call。
4. Tool Call 状态、输入、输出、时间正确。
5. 如果工具不需要审批，状态应为 completed。
6. 高风险工具仍应显示为需要审批，但不应被误执行。
