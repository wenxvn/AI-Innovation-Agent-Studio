## 4. 不可违反的开发原则

1. **不要推倒重来。** 当前 UI、路由、API 和数据模型如可用，继续增强。
2. **不要重做前端视觉。** UI 质量已经较高，只做必要交互和状态增强。
3. **不要继续扩大 mock。** mock 只能作为无 API Key 时的 fallback，必须显式标记 `mode: "mock"`。
4. **不要硬编码真实密钥。** 所有密钥只能来自 `.env`，并更新 `.env.example`。
5. **所有真实交互走 `/api/v1`。** 不要在页面组件中堆业务逻辑。
6. **高风险工具必须 Human Approval。** 写文件、删文件、执行 shell、改数据库、GitHub PR、部署、访问敏感文件等第一阶段不得静默真实执行。
7. **每个按钮必须有结果。** 要么真实 API，要么 disabled，要么 toast 显示功能开发中。
8. **每个阶段必须有验收命令。** 写完后给出运行命令、测试命令、访问路径和已知限制。
9. **保持向后兼容。** 迁移不能随意 drop 数据；不要删除用户已有文件。
10. **错误要可见。** 不允许静默吞错，前端要有 toast/error state，后端要有结构化错误。

## 6. Phase 1：LLM / Embedding Provider 抽象

### 6.1 目标

建立统一 Provider 层，让 Agent、RAG、Eval 不直接依赖某个厂商 SDK。

必须支持：

- 无 API Key 时使用 deterministic mock fallback。
- 有 API Key 时调用真实 LLM / Embedding。
- 所有响应必须记录 provider、model、mode、token_usage、latency_ms、error。
- 失败时不崩溃，返回结构化错误，并在前端显示。

### 6.2 建议新增或修改文件

后端：

```text
apps/api/app/core/config.py
apps/api/app/services/llm.py
apps/api/app/services/embeddings.py
apps/api/app/services/provider_types.py
apps/api/app/schemas/llm.py
apps/api/app/schemas/common.py
apps/api/app/tests/test_llm_provider.py
apps/api/app/tests/test_embedding_provider.py
.env.example
```

可选：

```text
apps/api/app/services/providers/openai_provider.py
apps/api/app/services/providers/anthropic_provider.py
apps/api/app/services/providers/deepseek_provider.py
apps/api/app/services/providers/qwen_provider.py
apps/api/app/services/providers/mock_provider.py
```

### 6.3 环境变量建议

更新 `.env.example`：

```env
# LLM Provider

LLM_API_KEY=

# Embedding Provider

```

### 6.4 LLM Provider 接口要求

实现等价接口：

```python
class LLMProvider(Protocol):
    async def generate(
        self,
        messages: list[dict],
        response_format: str | None = None,
        temperature: float = 0.2,
        max_tokens: int | None = None,
        metadata: dict | None = None,
    ) -> LLMResult:
        ...
```

`LLMResult` 至少包含：

```json
{
  "content": "string",
  "raw": {},
  "provider": "openai|anthropic|deepseek|qwen|mock",
  "model": "string",
  "mode": "real|mock",
  "token_usage": {
    "input_tokens": 0,
    "output_tokens": 0,
    "total_tokens": 0
  },
  "latency_ms": 0,
  "finish_reason": "stop",
  "error": null
}
```

### 6.5 Embedding Provider 接口要求

实现等价接口：

```python
class EmbeddingProvider(Protocol):
    async def embed_texts(self, texts: list[str], metadata: dict | None = None) -> EmbeddingResult:
        ...
```

`EmbeddingResult` 至少包含：

```json
{
  "vectors": [[0.0]],
  "dimension": 1536,
  "provider": "openai|qwen|mock",
  "model": "string",
  "mode": "real|mock",
  "latency_ms": 0,
  "error": null
}
```

### 6.6 Mock fallback 要求

mock fallback 只能用于：

- 没有 API Key。
- 本地离线演示。
- 自动化测试。

要求：

- 响应必须稳定、结构化、可测试。
- 必须显式返回 `mode: "mock"`。
- 前端必须能看到“当前运行于 Mock 模式”。
- 文档必须说明 mock 与真实能力边界。

### 6.7 验收标准

- `.env` 中 `LLM_PROVIDER=mock` 时，Agent Run 可正常完成并标记 mock。
- 配置真实 Provider 后，Agent Run 调用真实模型。
- `.env` 中 `EMBEDDING_PROVIDER=mock` 时，reindex 可正常完成并标记 mock。
- 配置真实 Embedding 后，chunk 写入真实向量。
- Provider 错误会写入 run/error，不导致后端进程崩溃。
- pytest 覆盖 provider fallback、超时、错误和成功场景。

## 8. Phase 3：RAG / pgvector 真闭环

### 8.1 目标

让上传文档不只是保存和展示，而是真正进入可检索知识库。

流程：

```text
上传文件
→ 提取文本
→ 清洗文本
→ chunking
→ embedding
→ 写入 document_chunks.embedding
→ Agent Run 根据用户问题向量检索
→ 检索结果进入 Context Pack.retrieved_evidence
→ Agent 输出引用证据来源
```

### 8.2 数据库要求

检查当前模型。如果缺失，补齐：

```text
document_chunks
  id
  document_id
  project_id
  chunk_index
  content
  token_count
  embedding vector(EMBEDDING_DIMENSION)
  embedding_provider
  embedding_model
  embedding_mode
  source_page
  source_section
  metadata jsonb
  created_at
  updated_at
```

必须使用 pgvector 索引。可选：

```sql
CREATE INDEX IF NOT EXISTS idx_document_chunks_embedding
ON document_chunks
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 100);
```

如果项目使用 Alembic，必须写 migration，不要手动改库后不留迁移。

### 8.3 API 要求

确认或补齐：

```text
POST   /api/v1/projects/{project_id}/documents/upload
GET    /api/v1/projects/{project_id}/documents
GET    /api/v1/projects/{project_id}/documents/{document_id}
DELETE /api/v1/projects/{project_id}/documents/{document_id}
POST   /api/v1/projects/{project_id}/documents/{document_id}/reindex
GET    /api/v1/projects/{project_id}/documents/{document_id}/chunks
POST   /api/v1/projects/{project_id}/rag/search
```

  "filters": {
    "document_ids": [],
    "source_type": "document"

### 8.4 Chunking 要求

第一阶段可使用简单 chunking：

- chunk size：约 800-1200 中文字符或等价 token。
- overlap：100-200 字符。
- 保留 source_page / source_section。
- 空文本、重复 chunk 要过滤。

### 8.5 Context Pack 要求

Agent Run 的 `context_pack.retrieved_evidence` 必须包含：

```json
{
  "source_type": "document_chunk",
  "source_id": "chunk_id",
  "document_id": "document_id",
  "document_name": "赛题.pdf",
  "excerpt": "...",
  "score": 0.82,
  "confidence": 0.82,
  "is_stale": false,
  "is_conflicting": false
}
```

### 8.6 前端要求

Files 页面：

- 显示 embedding 状态。
- 显示 embedding provider / mode。
- 显示 chunk 数量。
- Reindex 后状态更新。

Context 页面：

- 显示 RAG evidence。
- 可展开 chunk 原文。
- 显示相似度 score。
- 显示来源文档。

Chat 页面：

- Agent 回复中展示“本次使用的证据”。
- 如果是 mock embedding，必须提示。

### 8.7 验收标准

- 上传 TXT/Markdown/PDF 后生成 chunks。
- Reindex 后 chunks 有 embedding。
- 调用 `/rag/search` 能返回与 query 相关的 chunk。
- Agent Run 的 Context Pack 包含检索证据。
- 前端能看到 evidence 来源。
- 无 API Key 时 mock embedding 可工作，但明确显示 mock。

## 9. Phase 4：Eval 真实评分

### 9.1 目标

Eval 不能再固定 85 分或随机分。必须基于 rubric、规则或 LLM-as-judge 给出可解释评分。

### 9.2 Eval 维度

至少支持：

```text
correctness
completeness
feasibility
innovation
engineering_quality
citation_quality
rag_groundedness
tool_call_success
hallucination_risk
security_risk
latency
cost
```

### 9.3 Eval 策略

实现三层策略：

1. **Rule-based Eval**：无 LLM Key 时也能计算基础指标。
2. **RAG-grounded Eval**：检查输出是否引用了 Context Pack 中的 evidence。
3. **LLM-as-judge Eval**：有 LLM Key 时，用独立 judge prompt 生成评分和改进建议。

### 9.4 Eval 输出结构

```json
{
  "overall_score": 82,
  "mode": "real|mock|rule_based",
  "provider": "openai|mock|rule",
  "model": "string",
  "dimensions": [
    {
      "name": "completeness",
      "score": 84,
      "reason": "覆盖了目标用户、核心功能和验收标准，但缺少风险说明。",
      "evidence": []
    }
  ],
  "strengths": [],
  "weaknesses": [],
  "risks": [],
  "action_items": [],
  "latency_ms": 0,
  "token_usage": {}
}
```

### 9.5 Rubric 文件

建议新增：

```text
evals/rubrics/default_agent_output.yaml
evals/rubrics/prd.yaml
evals/rubrics/architecture.yaml
evals/rubrics/pitch.yaml
apps/api/app/services/evals.py
apps/api/app/prompts/eval_judge.md
```

### 9.6 API 要求

确认或补齐：

```text
GET  /api/v1/projects/{project_id}/evals
POST /api/v1/projects/{project_id}/evals/run
GET  /api/v1/projects/{project_id}/evals/{eval_id}
```

`POST /evals/run` 应支持：

```json
{
  "run_id": "optional",
  "output_id": "optional",
  "rubric": "prd",
  "mode": "auto|rule|llm"
}
```

### 9.7 前端要求

Evals 页面：

- 手动运行评估按钮必须可用。
- 显示 overall score。
- 显示每个维度的 reason。
- 显示 action items。
- 显示 mode/provider/model。
- 固定 mock 分数必须删除。

Chat / Outputs 页面：

- 关联显示 Eval 结果。
- 支持对某个 output 重新评估。

### 9.8 验收标准

- 对同一 output 调用 eval，评分基于内容和 rubric，而不是固定值。
- 无 LLM Key 时 rule-based eval 正常工作。
- 有 LLM Key 时 judge prompt 返回结构化评分。
- Eval 结果写入数据库，刷新后不丢。
- 前端可查看评分解释和改进建议。

## 12. Phase 7：Tool Approval 工程化

### 12.1 目标

Tool Gateway 不能只是展示列表。它必须记录工具调用，并对高风险操作实施 Human Approval。

### 12.2 Tool Call 状态

```text
requested
waiting_approval
approved
rejected
running
succeeded
failed
cancelled
```

### 12.3 高风险工具策略

以下操作必须审批，不得自动执行：

- 写文件
- 删除文件
- 执行 shell 命令
- 修改数据库
- 访问敏感文件
- GitHub PR / push
- 部署服务
- 调用外部付费或不可逆 API

第一阶段可以只做审批和审计记录，不做真实危险执行。

### 12.4 API 要求

```text
GET  /api/v1/tools
GET  /api/v1/projects/{project_id}/tools/calls
POST /api/v1/projects/{project_id}/tools/calls/{tool_call_id}/approve
POST /api/v1/projects/{project_id}/tools/calls/{tool_call_id}/reject
POST /api/v1/projects/{project_id}/tools/calls/{tool_call_id}/execute
```

审批请求：

```json
{
  "reason": "用户确认允许读取已上传文档并生成摘要"
}
```

拒绝请求：

```json
{
  "reason": "不允许执行 shell 命令"
}
```

### 12.5 前端要求

Tools 页面：

- 显示 pending approvals。
- 显示风险等级。
- 显示 tool input / output / error。
- Approve / Reject 按钮真实调用 API。
- 审批后状态刷新。

Chat / Inspector：

- 当 Agent Run 进入 `waiting_approval`，显示审批卡片。
- 审批后可继续运行或生成“需要重新运行”的提示。

### 12.6 验收标准

- 高风险 tool call 会进入 waiting_approval。
- Approve / Reject 改变数据库状态。
- Trace 记录审批事件。
- 前端状态刷新正确。
- 不会静默执行危险操作。
