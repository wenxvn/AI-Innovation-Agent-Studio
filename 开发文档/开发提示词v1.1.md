## 2. 总体开发原则

请严格遵守以下原则：

1. **先审计，再改代码。**  
   开始前先阅读目录结构、package、requirements、API、页面和 mock-data，不要直接大面积重写。

2. **保留现有高质量 UI 风格。**  
   不要把现有界面改成普通 admin dashboard。默认浅色模式要有 Cursor / Linear / Langfuse / Vercel 风格，深色模式保留科技感。

3. **不要继续扩大 mock。**  
   可以保留 demo seed 数据，但必须通过数据库/API 提供，而不是继续在页面里硬编码。

4. **后端 API 优先统一。**  
   所有前端真实交互必须走 `/api/v1` 后端 API。不要让业务逻辑散落在页面组件里。

5. **每个按钮必须有结果。**  
   要么接真实 API，要么 disabled 并显示“功能开发中”，不要留无响应按钮。

6. **优先打通闭环，不追求一次性做完所有高级能力。**  
   先做：项目 CRUD → 文件上传 → 解析/索引状态 → Agent Run 记录 → Outputs 保存 → Eval 简单评分。

7. **高风险工具不得真实执行危险操作。**  
   写文件、删除文件、执行 shell、修改数据库、提交 GitHub PR、部署服务等必须进入 Human Approval，第一阶段只做审计记录和审批状态，不做真实危险执行。

8. **所有新增功能都要有验收方式。**  
   每完成一个阶段，给出运行命令、访问路径、测试方法和已知限制。

9. **不要删除用户已有文件。**  
   迁移和重构要尽量兼容现有结构。

10. **最终必须更新文档。**  
   至少更新 `README.md`、`.env.example`、`docs/API_CONTRACT.md`、`docs/ROADMAP.md` 或 `PROJECT_STATUS_REPORT.md`。

## 12. Phase 8：RAG 最小闭环

### 12.1 目标

上传文档后，Agent 执行任务时可以检索相关 chunk，并进入 Context Pack。

### 12.2 RAG 流程

```text
上传文件
→ 解析文本
→ 切分 chunk
→ 生成 embedding
→ 保存到 pgvector / fallback vector
→ 用户提问或 Agent 执行任务
→ 检索相关 chunk
→ 构建 Context Pack
→ 生成带来源的回答或产物
```

### 12.3 第一阶段允许的降级策略

如果没有 embedding API Key：

- 使用 deterministic hash embedding 或简单关键词检索作为 fallback。
- UI 明确显示 `embedding_mode: fallback`。
- API 结构保持可替换真实 embedding provider。

### 12.4 Context Pack

每次 Agent 执行前构建：

```json
{
  "task": "",
  "goal": "",
  "current_stage": "",
  "constraints": [],
  "project_state": {},
  "relevant_memory": [],
  "retrieved_evidence": [],
  "tool_results": [],
  "decisions_so_far": [],
  "risks": [],
  "do_not_do": [],
  "expected_output": "",
  "output_schema": {}
}
```

每条 evidence 标记：

```text
source_type
source_id
filename
chunk_id
confidence
created_at
is_stale
is_conflicting
```

### 12.5 验收标准

- 上传文档后生成 chunks。
- Agent Run 可以检索 chunks。
- Context 页面显示 retrieved evidence。
- Agent 输出中能展示来源文件和 chunk。
- 删除文件后对应 chunks 不再被检索。
